Chapter 01

Object-Oriented Programming & Entities

In OOP, a class models a real-world concept. In full-stack apps these classes become database entities managed by JPA/Hibernate.

🏛️

What is a Class in OOP?

▼

A class is a blueprint that describes an object's data (fields) and behaviour (methods). In business applications, classes typically mirror real-world entities:

  • Client → represents a customer
  • Commande → represents an order
  • Employé → represents an employee
@Entity
public class Client {
    @Id
    @GeneratedValue
    private Long id;
    private String nom;
    private String email;
    // getters & setters...
}
💡 The @Entity annotation tells JPA: "Create a table for this class in the database."
🔗

Why Do Entities Need Links?

▼

Entities don't live in isolation — they have logical relationships. Examples:

  • A Client places many Commandes
  • A Commande belongs to exactly one Client
  • A Passport belongs to one Person

These links must be translated into JPA/Hibernate — that's what associations are for. In the database they become foreign keys or join tables.

🌍 Real world: An e-commerce platform like Amazon maps this exactly: Customer → Orders → Products.
📋

Simple Method to Model an Association

▼

The course gives a 3-step method:

  1. Write the relation in plain French/English
    "A client places several orders. An order belongs to one client."
  2. Identify the number of objects on each side
    Client → N orders | Order → 1 client
  3. Choose the correct annotation
    → @OneToMany on Client, @ManyToOne on Commande
✅ Always start with a business sentence — it prevents cardinality mistakes!
Chapter 02

Unidirectional vs Bidirectional Associations

An association defines how two entity classes are connected. The key question: does one side know about the other, or both?

2-1

Unidirectional 👉

Only one class holds a reference to the other.

Example: Client has a list of Commande, but Commande does NOT hold a reference back to Client.

@Entity
public class Client {
    @OneToMany
    private List<Commande> commandes;
}

@Entity
public class Commande {
    // No reference to Client here
}
✅ Simpler to code
✅ Easier to explain
✅ Good when navigation is one-way
⚠️ Can't navigate from Commande → Client
⚠️ May cause extra JOIN tables
2-2

Bidirectional ↔️

Both classes know each other.

Example: Client knows its Commandes AND each Commande knows its Client.

@Entity
public class Client {
    @OneToMany(mappedBy="client")
    private List<Commande> commandes;
}

@Entity
public class Commande {
    @ManyToOne
    private Client client; // owns FK
}
✅ Navigate both ways
✅ Richer object model
✅ Efficient queries
⚠️ Requires mastery of the owner side
⚠️ Must use mappedBy correctly

🔑 Key Rule for Bidirectional

In a bidirectional relation you must identify:

  • Owner side (propriétaire) — the side that actually holds the foreign key in the DB. Usually the @ManyToOne side.
  • Inverse side — uses mappedBy to point to the owner's field name. This side does NOT generate a foreign key column.
Chapter 03

Cardinalities — The 4 JPA Annotations

Cardinality describes how many objects of one type relate to another. JPA provides four dedicated annotations.

1 → 1

@OneToOne

▼

One object corresponds to exactly one other object.

📌 Example: A Person has one Passport and a Passport belongs to one Person.

@Entity
public class Personne {
    @OneToOne
    private Passport passport;
}

@Entity
public class Passport {
    // optional: back-ref
    @OneToOne(mappedBy="passport")
    private Personne personne;
}

DB result: A personne_id foreign key column is added to the passport table (or vice versa, depending on the owner).

🌍 Real use: User → Profile, Employee → Contract
1 → N

@OneToMany

▼

One object can be linked to multiple objects on the other side.

📌 Example: One Client can place many Commandes.

@Entity
public class Client {
    @OneToMany(mappedBy="client")
    private List<Commande> commandes
        = new ArrayList<>();
}

DB result: The Commande table holds a client_id foreign key column.

🌍 Real use: Author → Books, Department → Employees
N → 1

@ManyToOne

▼

Multiple objects are linked to one single object.

📌 Example: Many Commandes belong to one Client.

@Entity
public class Commande {
    @ManyToOne
    @JoinColumn(name="client_id")
    private Client client;
}

DB result: A client_id FK column appears in the commande table. This is the owner side of a bidirectional relation.

🌍 Real use: Comment → Post, Product → Category
N → N

@ManyToMany

▼

Multiple objects on both sides can relate to each other.

📌 Example: An Étudiant follows many Cours, and a Cours is followed by many students.

@Entity
public class Etudiant {
    @ManyToMany
    @JoinTable(name="etudiant_cours",
      joinColumns=@JoinColumn(name="etudiant_id"),
      inverseJoinColumns=@JoinColumn(name="cours_id"))
    private Set<Cours> cours
        = new HashSet<>();
}

@Entity
public class Cours {
    @ManyToMany(mappedBy="cours")
    private Set<Etudiant> etudiants;
}

DB result: An intermediate join table etudiant_cours is created with two FK columns.

🌍 Real use: Student ↔ Courses, User ↔ Roles, Product ↔ Tags

Chapter 04

Java Collections in JPA Associations

When a relation contains multiple objects (1-N or N-N), you must store them in a Java collection. The choice of collection impacts business logic and data behaviour.

C

Collection

▼

What: The root Java interface for all collection types. It's the most generic.

Characteristics: General interface — no ordering guarantee, duplicates depend on implementation.

Use when: You want to stay flexible and don't yet know which specific type you'll need.

private Collection<Commande> commandes;
💡 Prefer a more specific type (List or Set) for clarity.
L

List

▼

What: An ordered, indexed collection. Allows duplicate elements.

Characteristics: Maintains insertion order. You can access elements by index.

Use when: The order of elements matters (e.g., ordered list of commands by date).

@OneToMany(mappedBy="client")
private List<Commande> commandes
    = new ArrayList<>();
🌍 Real use: Timeline of events, ordered steps, ranked results.
S

Set

▼

What: A collection with no duplicate elements.

Characteristics: No guaranteed order. Each object appears only once.

Use when: You want to prevent duplicate entries — e.g., a user can't enrol in the same course twice.

@ManyToMany
private Set<Cours> cours
    = new HashSet<>();
🌍 Real use: User roles, product tags, student enrollments.
M

Map

▼

What: A key-value store. Each key is unique and maps to a value.

Characteristics: Fast lookup by key. Less common in JPA associations but useful for element collections.

Use when: You need to access data by a specific key (e.g., phone numbers by type: "home", "work").

@ElementCollection
private Map<String, String> phones;
🌍 Real use: Config properties, locale translations, attribute maps.

⚠️ Important Best Practice

Always initialise your collections from the start to avoid NullPointerException:

// ✅ CORRECT — initialised
private List<Commande> commandes = new ArrayList<>();
private Set<Cours> cours = new HashSet<>();

// ❌ WRONG — causes NullPointerException on first access
private List<Commande> commandes;

In practice, List and Set are the most frequently used collections in JPA associations.

Chapter 05

Reading the Mapping — Who Owns the Foreign Key?

The notes describe the full mapping logic for the classic Client–Commande example. Let's decode it step by step.

🧠

Business Rules for Client–Commande

▼
  1. A client can exist without any orders.
  2. An order always belongs to exactly one client.
  3. Multiple orders can belong to multiple clients.

From these rules we derive:

  • Client side → @OneToMany
  • Commande side → @ManyToOne (owns the FK)
🗄️

What Happens in the Database?

▼

The Commande table contains a column client_id — the foreign key that links each order to its client.

🔑 id
nom
email
← FK
commande
🔑 id
montant
🔗 client_id

The client_id column in commande is the foreign key, confirming that Commande is the owner side.

🔄

ManyToMany — The Join Table

▼

For N–N (e.g., Étudiant ↔ Cours), there is no FK column in either entity table. Instead JPA creates an intermediate join table:

etudiant
🔑 id
nom
↔
etudiant_cours
🔗 etudiant_id
🔗 cours_id
↔
cours
🔑 id
titre
📐

Owner Side Rule

▼

The owner side is the one that actually contains the foreign key (or join table definition). Key facts:

  • In a OneToMany / ManyToOne pair → @ManyToOne is always the owner.
  • In a OneToOne → the side with @JoinColumn is the owner.
  • In a ManyToMany → the side with @JoinTable is the owner.
  • The inverse side uses mappedBy="fieldName" pointing to the owner's field.
⚠️ If you forget mappedBy, Hibernate creates an extra unwanted join table!

🎯 Complete Bidirectional Client–Commande Example

// ── Client.java ─────────────────────────────
@Entity
public class Client {
    @Id @GeneratedValue
    private Long id;
    private String nom;

    @OneToMany(mappedBy = "client", cascade = CascadeType.ALL)
    private List<Commande> commandes = new ArrayList<>();
}

// ── Commande.java ───────────────────────────
@Entity
public class Commande {
    @Id @GeneratedValue
    private Long id;
    private Double montant;

    @ManyToOne
    @JoinColumn(name = "client_id")  // ← Owner side
    private Client client;
}
